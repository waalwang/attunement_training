# DPO Synthetic Rejected Pipeline

**Date:** April 24, 2026
**Status:** Stage 1 implemented, Stage 2 implemented (pending SFT checkpoint)

---

## 1. Motivation

With both unweighted and weighted SFT jobs running, we need a DPO dataset
where rejected trajectories are realistic -- not random garbage, but the
kind of mediocre responses a partially-trained model would produce. The
approach combines real low-quality conversation branches from the comment
trees with synthetic text generated by the SFT checkpoint.

---

## 2. Initial Idea

Build DPO training data as another version of trajectory extraction:

- **Chosen chains:** high-quality chains similar to the SFT training data
- **Rejected chains:** a new approach -- find low-score candidates from
  the same comment trees sharing a common fork with the chosen chain

For the rejected branch, two requirements must be met simultaneously:

1. **Preserve authentically bad turns.** Keep original assistant turns that
   already have low scores. For high-score assistant turns (score >= 5),
   replace them with synthetic text generated by the SFT checkpoint so the
   whole trajectory is consistently low quality.

2. **Length matching.** The total character length of the rejected branch
   must match the chosen branch. If length is unmatched after assistant
   replacement, also replace some user-side turns with synthetic text.

---

## 3. Brainstorm & Design Decisions

### 3.1 Fork definition

A fork is the shared prefix of two conversation chains in a comment tree.
Forks can be multi-turn (not just the first user post). Allowing
variable-depth forks maximizes the number of pairable chosen/rejected
candidates and increases dataset volume.

The fork is identified by content: two chains share a fork when their
first K turns are identical and they diverge at turn K+1. The same post
can produce fork groups at multiple depths.

### 3.2 Chosen and rejected are fully divergent branches

The fork (shared prefix) is fixed. Everything after the fork -- both user
and assistant turns -- diverges completely between chosen and rejected.
These are two entirely different multi-turn trajectories stemming from the
same root. This means:

- Touching user-side turns in the rejected branch for length matching is
  harmless, because user turns are part of the divergent trajectory and
  are not counted in the loss (masked).
- This is trajectory-level DPO, not single-turn DPO.

### 3.3 Weighting in DPO

Standard DPO loss is a log-ratio contrast between chosen and rejected
policy probabilities. The signal comes from the gap between them, not
from weighting individual tokens. Per-token weights on the rejected side
would distort the contrast in ways that are not well-studied.

Decision: **no per-token weights on the rejected side.** The rejected
trajectory just needs to be a realistically bad trajectory. Weighting
can optionally be applied on the chosen side to upweight the turns we
most want the model to learn.

### 3.4 Scoring synthetic data

- **Attunement score:** computable for synthetic data using the existing
  feature-based tone scorer (scripts/tone_scorer.py).
- **Upvote score:** not applicable to synthetic data.
- **Resolution:** scores are only needed to select which chains/turns are
  low-quality (done using existing scores in the raw data) and to decide
  which assistant turns to replace (score >= threshold). Synthetic
  replacement turns do not need a score -- their purpose is to degrade a
  high-scoring turn into a mediocre one.

### 3.5 Matching strategy

1. Start from the raw JSON files containing full comment trees.
2. Extract chosen chains (the best, same as SFT training).
3. For each chosen chain, walk the tree to find rejected candidates at
   different fork depths.
4. Match by turn count (within +/-2), collect all rejected candidates.
5. Sort candidates by score ascending, pick the first one where:
   - rejected score < chosen score
   - rejected is not excessively longer than chosen:
     (rej_chars - cho_chars) / max(rej_chars, cho_chars) <= 0.3
6. Length-aware fallback: if the best candidate (lowest score) leaves
   rejected shorter than chosen by > 1000 chars, scan alternatives
   within score delta <= 10 and swap to whichever has the closest
   length to chosen.
7. If the chosen branch is longer than rejected by > 30%, Stage 2
   expands the rejected branch with synthetic text. If still unmatched
   after synthesis, the pair is dropped.

### 3.6 Search-from-middle strategy

To prevent the extractor from producing "heavy-fork-tiny-branch" pairs
(long shared prefix with a trivially short divergent branch), the fork
search starts from the middle of the chosen chain and spirals outward.

For a chosen chain of length N, fork candidates are tried in order:
N/2, N/2-1, N/2+1, N/2-2, N/2+2, ...

This prioritizes balanced fork/branch ratios. Candidates from all fork
depths are collected, sorted by rejected score ascending. The first
one passing score and length constraints wins. If the best candidate
leaves a > 1000 char gap, nearby candidates (score delta <= 10) are
checked for a closer length match.

## 4. Pipeline Architecture

Two-stage pipeline: Stage 1 is pure extraction (CPU, fast, concurrent
workers). Stage 2 is synthetic generation (GPU-bound, runs after SFT
training completes).

### Stage 1: Extract DPO pairs

**Files:**
- `../crawler/conversation_crawler/processors/dpo_pair_extractor.py`
  -- core extraction algorithm
- `../crawler/conversation_crawler/scripts/extract_dpo_pairs.py`
  -- concurrent runner (same pattern as extract_sft_chains.py)

**Algorithm:**
```
For each thread (raw JSON):
  1. Build comment tree (filter noise, enrich reply counts)
  2. Extract chosen chains (high-quality, same as SFT extractor)
     - min_turn_score >= 2, min_chain_depth >= 3
     - Prefix-deduplicated, sorted by total score
  3. For each chosen chain:
     a. Search fork points from middle outward
     b. At each fork, find siblings of the chosen path's next node
     c. From each sibling, extract greedy best chain downward
     d. Check turn count within +/-2 of chosen branch
     e. Collect all valid candidates (rej_score < cho_score,
        length excess <= 0.3)
     f. Sort by rejected score ascending, pick best
     g. If best leaves > 1000 char gap (chosen longer), scan
        alternatives within score delta <= 10 for closer length
  4. Deduplicate: each rejected branch used at most once per thread
  5. Output: fork + chosen_branch + rejected_branch + metadata
```

**Output parquet columns:**
- fork, chosen_branch, rejected_branch (JSON strings)
- source, subreddit, post_id, post_url
- post_score, fork_depth, chosen_depth, rejected_depth
- chosen_score, rejected_score, score_delta
- chosen_chars, rejected_chars
- score_type

### Stage 2: Synthetic generation

**File:**
- `scripts/generate_dpo_rejected.py`

**Algorithm:**
```
For each pair from Stage 1:
  1. Parse fork, chosen_branch, rejected_branch
  2. Asymmetric length check (safety net for Stage 1):
     - If rejected is longer than chosen by > 30%
       ((rej-cho)/max(rej,cho) > 0.3): drop pair
     - If chosen is longer: proceed to expansion below
  3. For each assistant turn in rejected_branch:
     - If score >= threshold (default 5): replace with SFT-generated text
       Context = fork + rejected turns up to that point
       Target length = original turn's character count
  4. Length expansion (only when rejected is shorter than chosen):
     - Compute (cho-rej)/max(cho,rej)
     - If ratio exceeds tolerance (default 30%):
       - Find user turns in rejected_branch
       - Regenerate with adjusted target length to close the gap
       - User generation uses a system prompt for role-playing
  5. Final length check: if still unmatched after synthesis, drop pair
  6. Output: final rejected_branch with synthetic flag per turn
```

**Context for synthetic generation:**
- Assistant replacement: context = fork + all rejected branch turns
  up to that point. Each synthetic assistant turn sees the conversation
  history it's "responding to."
- User replacement: same pattern -- fork + preceding turns on the
  rejected branch. Plus a role-playing system prompt so the model
  generates as a user, not an assistant.

Synthetic turns are contextually coherent within the rejected
trajectory -- a plausible continuation, not isolated text.

**Output parquet columns:**
- Same as Stage 1, plus:
- asst_replacements, user_replacements (counts)
- length_matched (bool)

## 5. Usage

### Stage 1: extraction (run from crawler directory)

```bash
cd ../crawler/conversation_crawler

# Reddit (~3600 batches, 8 workers)
python scripts/extract_dpo_pairs.py \
    --raw-dir data/raw \
    --output-dir data/dpo_pairs \
    --source reddit \
    --workers 8

# HN (~106 batches, append to same output)
python scripts/extract_dpo_pairs.py \
    --raw-dir data/raw_hn \
    --output-dir data/dpo_pairs \
    --source hn \
    --workers 4 \
    --append

# Quick test (2 batches only)
python scripts/extract_dpo_pairs.py \
    --raw-dir data/raw \
    --output-dir data/dpo_pairs_test \
    --source reddit \
    --workers 1 \
    --limit 2
```

Key parameters:
- `--turn-tolerance 2` (default): chosen/rejected branch turn count can
  differ by up to 2
- `--min-branch-depth 2` (default): branches shorter than 2 turns are
  skipped
- `--min-turn-score 2` (default): chosen chain quality floor
- `--max-chains-per-post 20` (default): cap chosen chains per thread
- `--max-length-excess 0.3` (default): drop pairs where rejected is
  longer than chosen by more than 30%

### Stage 2: synthetic generation (run from attunement_training)

```bash
# After SFT training completes:
python scripts/generate_dpo_rejected.py \
    --input-dir ../crawler/conversation_crawler/data/dpo_pairs \
    --checkpoint outputs/sft/checkpoint-XXXX \
    --output-path data/dpo_final.parquet \
    --score-threshold 5 \
    --length-tolerance 0.15

# Test on small subset:
python scripts/generate_dpo_rejected.py \
    --input-dir ../crawler/conversation_crawler/data/dpo_pairs \
    --checkpoint outputs/sft/checkpoint-XXXX \
    --output-path data/dpo_test.parquet \
    --limit 50
```

Key parameters:
- `--score-threshold 5`: assistant turns with score >= 5 get replaced
- `--length-tolerance 0.30`: max 30% char-length difference allowed;
  rejected-too-long pairs are dropped, chosen-too-long pairs are expanded
- `--temperature 0.7`, `--top-p 0.9`: generation sampling params
- `--bf16`: use bfloat16 for inference (default on)

## 6. Volume Estimate

From a test on 1 batch (20 Reddit threads):
- 20 threads -> 100 DPO pairs (5.0 pairs/thread average)
- Extrapolated: ~3627 Reddit batches x 20 posts x 5 pairs = ~362K pairs
- Plus HN data: ~106 batches

Actual yield will vary based on tree structure and fork availability.

## 7. Data Flow

```
raw JSON files (Reddit/HN comment trees)
    |
    v
[Stage 1] extract_dpo_pairs.py (concurrent, CPU)
    |
    v
data/dpo_pairs/*.parquet (raw pairs with real rejected branches)
    |
    v
[Stage 2] generate_dpo_rejected.py (GPU, SFT checkpoint)
    |
    v
data/dpo_final.parquet (synthetic-enhanced rejected branches, length-matched)
    |
    v
[Optional] tone_scorer.py (compute attunement scores on synthetic turns)
    |
    v
DPO training
```
